# 운동탭 테스트모드 진입 화면 개편안

작성일: 2026-05-07
범위: 운동탭에서 테스트모드로 진입했을 때 처음 보이는 화면
산출물: `mockups/test-mode-entry-apple-toss-figma.html`

## 1. 한 줄 방향

테스트모드 첫 화면은 "추천 목록"이 아니라 **오늘 들어야 할 무게를 바로 조정하고 시작하는 수행 보드**가 되어야 한다.

현재 화면은 사용자가 테스트모드에 들어오자마자 다음을 해석하게 만든다.

- 오늘 부위를 먼저 고르라는 게 설정인지, 운동 시작인지 불명확하다.
- 벤치마크, 성장판, 보강 추천, 계획 조정이 같은 우선순위로 보인다.
- 무게와 트랙이 정보처럼 보여서 "내가 바로 바꿀 수 있다"는 감각이 약하다.
- 일반/프로/테스트 모드 전환 UI가 첫 화면의 공간을 많이 차지한다.

개편안은 이 흐름을 바꾼다.

> 진입 즉시 오늘 날짜, Week, 트랙, 첫 종목의 무게가 보인다. 사용자는 숫자를 보고, 필요하면 바꾸고, 바로 시작한다.

## 2. 디자인 원칙

### 2.1 Apple 관점: 집중

첫 화면의 주인공은 하나다.
운동 직전 사용자가 가장 먼저 알아야 하는 것은 "오늘 몇 kg으로 시작하는가"다.

- 첫 viewport 안에 큰 숫자형 무게 컨트롤을 둔다.
- 모드 설명, 프레임워크 설명, 추천 근거는 첫 화면에서 강등한다.
- 오늘 하지 않을 정보는 bottom sheet나 상세 화면으로 보낸다.

### 2.2 Toss 관점: 결정 피로 제거

추천 카드를 수락/수정/숨김하는 구조는 금융 앱으로 치면 송금 전 승인 카드를 여러 장 읽게 하는 것이다.
토스식 해법은 "합리적 기본값을 이미 넣어두고, 숫자만 바꾸게 하는 것"이다.

- CTA는 화면 하단 1개로 제한한다.
- 각 종목 row에는 `+/-`와 무게 숫자만 둔다.
- "왜 이 무게인가"는 작은 근거 pill 또는 상세 링크로 감춘다.

### 2.3 Figma 관점: 컴포넌트화

현재 테스트모드 화면은 카드, 칩, 추천, 보드가 각각 다른 문법을 쓴다.
새 화면은 다음 컴포넌트만 반복한다.

- `EntryHeader`: 뒤로, 날짜/Week, 계획 메뉴
- `TrackSegment`: 볼륨/강도 전환
- `BodyPartRail`: 오늘 부위 선택
- `HeroLift`: 오늘 첫 종목과 주 무게
- `PrescriptionRow`: 남은 종목 리스트
- `AssistBlock`: 마지막 10분 보강
- `StickyStartBar`: 시작 CTA

## 3. 화면 구조

### 3.1 진입 상태 A: 부위 미선택

목표는 "선택을 요구하는 설정 화면"이 아니라 "오늘 시작점을 제안하는 화면"이다.

첫 화면 구성:

1. 상단: `오늘 · Week 2/6`
2. 짧은 문장: `오늘은 가슴/삼두로 시작할 수 있어요`
3. 부위 rail: 가슴, 등, 하체, 어깨, 팔, 코어
4. 선택 전 preview: 첫 벤치마크 2개만 흐리게 표시
5. 하단 CTA: `가슴/삼두로 시작`

사용자가 부위를 바꾸면 CTA와 preview가 즉시 바뀐다.

### 3.2 진입 상태 B: 부위 선택 후

목표는 수행 보드다.

첫 viewport 구성:

1. 상단: 뒤로, 날짜, Week, 계획 메뉴
2. 트랙: 볼륨 / 강도 segmented control
3. 핵심 카드: 첫 종목, 무게, `-2.5 / +2.5`, 세트/반복
4. 진행 rail: Week 2에서 목표 Week 6까지
5. 남은 종목 2-3개: 리스트 row
6. 하단 고정 CTA: `타이머 시작`

### 3.3 진입 상태 C: 이미 운동 중

목표는 재진입 복구다.

- 상단에 `진행 중 · 38:12` 상태를 노출한다.
- 첫 CTA는 `운동으로 돌아가기`다.
- 수행 완료된 종목은 접고, 다음 미완료 종목을 hero로 올린다.

## 4. 정보 위계

| 우선순위 | 정보 | 표시 방식 |
|---|---|---|
| 1 | 오늘 첫 종목과 무게 | 큰 숫자, inline stepper |
| 2 | 볼륨/강도 트랙 | 상단 segmented control |
| 3 | 세트/반복/RIR | 숫자 아래 보조 텍스트 |
| 4 | Week 진행률 | 얇은 rail |
| 5 | 남은 종목 | compact list row |
| 6 | 보강 추천 | 마지막 10분 block |
| 7 | 근거/설정 | 상세 sheet |

## 5. 카피 원칙

첫 화면에서 지울 표현:

- "추천을 수락하세요"
- "프레임워크"
- "전담 코치"
- "비중 낮음"
- "초안"

대체 표현:

- `오늘 시작 무게`
- `계획보다 2.5kg 낮춰도 괜찮아요`
- `지난 성공값 기준`
- `마지막 10분`
- `계획 조정`

운동 직전에는 설명보다 확신이 필요하다.

## 6. 스타일 원칙

- 카드 radius는 8px 기준으로 맞춘다.
- 색은 회색조 중심, 토마토는 CTA/진행/오늘 표식에만 쓴다.
- 부위별 색상은 쓰지 않고 텍스트와 선택 상태로 구분한다.
- 이모지는 쓰지 않는다.
- 숫자는 tabular-nums를 적용해 `+/-` 조정 시 레이아웃이 흔들리지 않게 한다.
- 버튼은 최소 44px 터치 타깃을 지킨다.

## 7. 현재 코드 반영 위치

실제 구현 시 손댈 파일:

- `workout/expert/max.js`
  - `_renderMaxTodayMajorGate()`를 smart entry gate로 재구성
  - `renderMaxCard()`에서 모드 선택 stack을 첫 화면 밖으로 축소
  - 부위 선택 후 바로 dashboard가 아니라 hero lift 중심 board로 렌더
- `workout/expert/max-cycle-render.js`
  - `_renderV4Lift()`를 `HeroLift`와 `PrescriptionRow`로 분리
  - `renderMaxCycleDashboard()`의 추천 패널 위치를 하단 보강 block으로 이동
- `expert-mode.css`
  - `wt-v4-*` 토큰 정리
  - 카드 radius, type scale, touch target, sticky CTA 정리
- 테스트
  - plan editor HTML의 `onclick=` 금지 테스트 유지
  - 트랙 변경, 부위 변경, 무게 stepper가 draft를 보존하는지 추가 검증

`expert-mode.css`는 `sw.js`의 `STATIC_ASSETS`에 포함되어 있으므로 실제 CSS를 수정하는 단계에서는 `CACHE_VERSION`도 함께 올려야 한다.

## 8. 구현 단계

### Phase 1: 진입 화면만 교체

- 부위 미선택 gate를 smart start 형태로 교체
- 상단 모드 선택 stack 제거 또는 compact menu로 이동
- 기존 데이터 계산 로직은 유지
- HTML 구조와 CSS만 최소 변경

완료 기준:

- 테스트모드 진입 후 첫 viewport에 오늘 날짜, Week, 부위 선택, CTA가 모두 보인다.
- 이모지와 다중 추천 CTA가 첫 viewport에 없다.
- 부위 선택 전/후 CTA 문구가 바뀐다.

### Phase 2: 수행 보드 재배치

- 첫 벤치마크를 hero lift로 승격
- 나머지는 compact prescription row로 낮춤
- 무게 `- / +` 조정은 row 안에서 바로 가능하게 함
- 트랙 segmented control이 모든 row의 계획값에 영향을 준다는 피드백을 즉시 보여줌

완료 기준:

- 진입 후 0.3초 안에 오늘 첫 무게를 읽을 수 있다.
- 트랙 변경과 무게 조정이 화면 이동 없이 가능하다.
- CTA는 하단 1개다.

### Phase 3: 보강 추천 강등

- 추천 카드는 첫 화면에서 제거
- `마지막 10분` block에 1-2개만 노출
- 상세 근거는 sheet로 이동

완료 기준:

- 추천 수락/수정/숨김 CTA가 첫 화면에 반복되지 않는다.
- 보강 추천은 운동 시작을 방해하지 않는다.

## 9. 리스크와 방어선

- 리스크: 기존 사용자가 "추천 카드가 사라졌다"고 느낄 수 있다.
  - 방어선: `마지막 10분` block과 계획 메뉴 안에 추천 상세를 남긴다.
- 리스크: 무게 조정이 draft cycle을 덮어쓸 수 있다.
  - 방어선: Max V4 Draft Preservation Rule에 맞춰 DOM 값을 draft로 먼저 흡수한다.
- 리스크: 모달 sheet의 `stopPropagation()` 때문에 버튼이 안 먹을 수 있다.
  - 방어선: `.wt-v4-sheet` capture binding 또는 직접 binding을 유지한다.

## 10. 목업 확인 포인트

HTML 목업은 다음 두 상태를 보여준다.

- 왼쪽: 테스트모드 첫 진입, 부위 선택 전 smart gate
- 오른쪽: 부위 선택 후 오늘 수행 보드

파일: `mockups/test-mode-entry-apple-toss-figma.html`

목업에서 확인할 것:

- 첫 화면이 설정처럼 보이지 않고 운동 시작 화면처럼 보이는가
- 무게가 정보가 아니라 컨트롤처럼 보이는가
- 추천이 운동 시작을 방해하지 않는가
- 토마토 색이 의미를 하나만 말하는가
