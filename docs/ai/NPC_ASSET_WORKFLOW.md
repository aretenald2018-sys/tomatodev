# NPC 아트에셋 제작 워크플로우

## 적용 대상

홈탭 라이프존의 NPC, 캐릭터, 전구 상호작용, 퀘스트/대화 모달, NPC 전용 공간/소품 스프라이트를 추가하거나 수정하는 모든 작업에 적용한다.

앱 코드나 자산을 수정하기 전에 이 문서를 먼저 읽고, 계획 문서에 이 문서의 체크리스트 답을 남긴다.

## 기본 산출물

새 NPC는 기본적으로 두 벌의 아트에셋이 필요하다.

1. 홈탭 배치용 스프라이트
   - 홈 라이프존 공간 안에 실제로 서 있거나 앉아 있는 작은 아이소메트릭 스프라이트다.
   - 기존 방의 바닥 각도, 조명, 가구 스케일, 픽셀/반픽셀 질감과 맞춰야 한다.
   - 캐릭터 이름은 이미지에 굽지 않고 DOM 이름표로 렌더한다.

2. 퀘스트/대화 모달용 아트에셋
   - 전구 버튼 클릭 후 모달에 나오는 큰 좌상, 반신, 앉은 자세, 또는 전신 아트다.
   - 홈탭 스프라이트를 억지로 확대하지 않는다.
   - 홈 스프라이트와 같은 인물로 인식되어야 하지만, 모달의 자세와 크기는 모달 전용으로 다시 만든다.

필요하면 세 번째 산출물을 만든다.

3. NPC 전용 공간/소품 스프라이트
   - NPC가 관할하는 공간이 필요할 때 만든다.
   - 예: 미란다의 옷 행거, 의상, 선반, 전신거울 패션 코너.
   - 기존 베이스룸 전체를 새로 그리기 전에 투명 PNG overlay로 자연스럽게 덮는 방식을 우선 검토한다.

## 홈탭 배치 원칙

1. 기존 홈탭 베이스룸이 기준이다.
   - 좌표계는 현재 라이프존 기준 `1672x1672`을 우선 유지한다.
   - 사용자가 명시하지 않았으면 새 방, 새 트랙, 새 배경을 만들지 않는다.
   - 이미 만들어진 러닝트랙, 책상, 운동기구, NPC, 캐릭터를 기준으로 배치한다.

2. 공간 속에 녹아들어야 한다.
   - NPC가 카드, 스티커, UI blob, 추상 도형처럼 보이면 실패다.
   - NPC가 트랙, 기구, 책상, 가구 위에 떠 있거나 어색하게 겹치면 실패다.
   - NPC가 관할하는 소품/공간과 함께 읽혀야 한다. 공간 밖에 캐릭터만 붙이면 실패다.

3. 각도와 시선은 방의 원근을 따른다.
   - 캐릭터가 바라보는 방향은 바닥 타일, 의자/책상 방향, 주변 소품의 각도와 맞춘다.
   - 단순히 `scaleX(-1)`로 뒤집기 전에, 뒤집은 결과가 공간 안에서 자연스러운지 합성 미리보기로 확인한다.
   - 앉은 NPC는 실제 앉을 물체나 바닥 위치가 있어야 한다.

4. 임의 스케일 변경을 금지한다.
   - “대충 작게/크게” 조정하지 않는다.
   - 기존 트레이너, 미란다, 라이프존 actor와 비교한 기준 px를 계획 또는 리뷰에 기록한다.
   - 홈용 스프라이트, 전구, 이름표, 모달용 아트는 각각 별도 크기 기준을 갖는다.

## Imagegen 프롬프트 기준

imagegen으로 NPC 자산을 만들 때 프롬프트에는 최소한 다음을 포함한다.

1. 기존 홈탭 아이소메트릭 픽셀아트/반픽셀아트 화풍
2. 홈 배치용인지 모달용인지 명확한 용도
3. 캐릭터의 자세, 시선, 바닥 방향, 앉을 물체 또는 서 있을 위치
4. 투명 배경 또는 chroma-key 배경
5. 텍스트, 로고, 말풍선, 추상 도형, UI 카드, 배경 장식 금지
6. 홈 배치용이면 작은 스프라이트 구도, 모달용이면 모달 전용 자세

실존 인물이나 저작물 모티브는 직접 복제가 아니라 분위기 기준으로만 쓴다.

예: “검은 선글라스와 나이 든 패션 편집장 분위기의 원본 NPC”는 가능하지만, 특정 배우/캐릭터의 정확한 얼굴 복제, 로고, 상표 재현은 피한다.

## 후처리 기준

1. 최종 자산은 RGBA 투명 PNG로 저장한다.
2. 알파 가장자리, 잔여 chroma-key, 잘린 팔다리, 불필요한 배경 조각을 확인한다.
3. 이미지 안 글자는 신뢰하지 않는다. 이름, 안내문, 상태 문구는 DOM 텍스트로 처리한다.
4. PNG header 크기와 `colorType: 6`을 테스트에 넣는다.
5. 자산 파일은 원본 성격에 따라 분리한다.
   - 홈/모달 UI 자산: `assets/home/life-zone/ui/`
   - actor animation sprite: `assets/home/life-zone/sprites/`
   - 문서용 초안: `docs/ai/art-drafts/`

## UI/이벤트 기준

1. NPC 이름표
   - 트레이너처럼 DOM 이름표를 사용한다.
   - 홈탭 NPC 이름은 `lz-nameplate--npc` 계열로 표시한다.
   - 이미지에 이름을 굽지 않는다.

2. 전구 상호작용
   - 전구는 기존 `npc-quest-bubble.png`를 우선 재사용한다.
   - 클릭 가능한 요소는 `button`이어야 한다.
   - `aria-label`, `title`, `data-lz-action`을 명확히 둔다.
   - 이벤트는 기존 `life-zone:npc-quest`를 유지하고 `detail: { npc: '<id>' }`로 분기한다.

3. 모달
   - 모달용 아트에셋은 홈 배치용 스프라이트와 별도로 둔다.
   - sheet 내부에 `event.stopPropagation()`이 있으면 버튼은 모달 모듈 안에서 직접 바인딩한다.
   - 새 inline `onclick`을 만들지 않는다.

## 서비스워커와 검증

1. `sw.js` `STATIC_ASSETS`에 포함되는 파일을 추가/수정하면 반드시 `CACHE_VERSION`을 bump한다.
2. `www/`는 직접 수정하지 않는다.
3. 런타임 자산은 `node scripts/verify-runtime-assets.mjs`로 검증한다.
4. 최소 검증:
   - `node --check` 대상 JS 파일
   - 관련 `node --test` 파일
   - `node scripts/verify-runtime-assets.mjs`
   - `git diff --check`
   - Dashboard3 배포가 필요한 경우 `npm.cmd run verify:deploy -- https://aretenald2018-sys.github.io/dashboard3/ <commit>`

인증 세션이 없어 실제 홈 UI 클릭 flow를 확인하지 못하면, 가능한 정적/배포 자산 검증을 끝낸 뒤 `not verified yet`과 정확한 blocker를 기록한다.

## 구현 전 체크리스트

NPC 작업 계획에는 다음 답이 있어야 한다.

1. 이 NPC의 홈 위치는 기존 방의 어느 좌표/공간인가?
2. 같은 공간에 있는 가구, 운동기구, 트랙, 캐릭터와 겹치는가?
3. 홈 배치용 스프라이트와 모달용 아트에셋을 각각 만들었는가?
4. NPC 전용 공간/소품 overlay가 필요한가?
5. 캐릭터 시선과 자세가 홈탭 원근과 맞는가?
6. 기존 NPC/actor와 비교한 스프라이트 크기 기준은 무엇인가?
7. 이름표와 전구가 이미지가 아니라 DOM 구조로 들어가는가?
8. 새 PNG/JS/CSS가 `sw.js` `STATIC_ASSETS`에 포함되는가?
9. `CACHE_VERSION` bump가 필요한가?
10. 배포 후 어떤 URL, 어떤 UI 상태, 어떤 HTTP 상태가 완료 증거인가?

이 체크리스트에 답하지 않은 상태로 앱 코드나 아트에셋을 만들지 않는다.
