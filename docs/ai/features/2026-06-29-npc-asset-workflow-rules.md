# NPC 아트에셋 제작 워크플로우 규칙 문서화 계획

## 상태

- 상태: `implemented`
- 요청일: 2026-06-29
- 목적: 앞으로 홈탭 NPC/캐릭터를 추가할 때 반복되는 시행착오를 줄이기 위해, 작업 전 필수로 읽는 문서에 NPC 아트에셋 제작 기준을 고정한다.

## 배경

오늘 홈 라이프존의 트레이너, 러닝 캐릭터, 미란다 NPC를 조정하면서 다음 문제가 반복됐다.

1. 홈탭 공간에 실제로 녹아드는 스프라이트와 모달용 아트에셋이 구분되지 않았다.
2. 홈탭 인테리어의 원근/시선/바닥 위치와 맞지 않는 캐릭터가 배치됐다.
3. 캐릭터 크기와 위치가 기존 NPC/운동기구/트랙과 비교되지 않아 혼자 커 보이거나, 트랙 위에 올라탄 것처럼 보였다.
4. 새 공간을 만들어야 할 때 기존 베이스룸을 임의로 갈아엎거나 이상한 도형/플레이스홀더처럼 보이는 에셋이 섞일 위험이 있었다.
5. 사용자에게 보이는 이름, 전구 상호작용, 모달 진입, 서비스워커 캐시, PNG 검증이 매번 따로 판단됐다.

## 문서화 대상 결정

1. `AGENTS.md`

   - NPC/캐릭터/라이프존 아트 요청이 들어오면 `docs/ai/NPC_ASSET_WORKFLOW.md`를 먼저 읽으라는 진입점을 둔다.

2. `docs/ai/NPC_ASSET_WORKFLOW.md`

이 구조를 쓰면 `AGENTS.md`가 비대해지지 않고, 다음 작업자가 반드시 읽어야 하는 세부 기준은 별도 문서에서 유지할 수 있다.

## NPC 기본 산출물 계약

새 NPC를 만들 때 기본 산출물은 최소 2벌이다.

1. 홈탭 공간 배치용 스프라이트
   - 홈 라이프존 내부에 실제로 서 있거나 앉아 있는 작은 아이소메트릭 스프라이트다.
   - 기존 베이스룸의 각도, 바닥 방향, 조명, 픽셀/반픽셀 느낌과 맞아야 한다.
   - 캐릭터 이름은 이미지에 굽지 않고 DOM 이름표로 표기한다.

2. 퀘스트/대화 모달용 아트에셋
   - 전구 버튼 클릭 후 모달에 등장하는 큰 좌상/앉은 자세/반신 아트다.
   - 홈 스프라이트와 같은 인물로 인식되어야 하지만, 홈 배치용 스프라이트를 억지로 확대해 쓰지 않는다.
   - 모달의 시선, 자세, 크기는 기존 트레이너/미란다 모달과 비교해 정한다.

필요하면 3번째 산출물을 만든다.

3. NPC 전용 공간/소품 스프라이트
   - NPC가 관할하는 영역이 필요할 때 사용한다.
   - 예: 미란다는 좌측 하단을 옷 행거, 의상, 선반, 거울이 있는 패션 코너로 만들었다.
   - 기존 베이스룸 전체를 재생성하기보다, 먼저 투명 PNG overlay로 자연스럽게 덮는 방식을 검토한다.

## 홈탭 배치 원칙

1. 기존 홈탭 베이스룸을 우선 기준으로 삼는다.
   - 좌표계는 현재 라이프존 기준 `1672x1672`을 우선 유지한다.
   - 별도 방, 별도 트랙, 별도 배경을 새로 만들지 않는다. 사용자가 명시한 경우만 예외다.

2. 배치 전 반드시 기준 비교를 한다.
   - 트레이너, 기존 캐릭터, 러닝트랙, 책상, 운동기구, 가구와 크기를 비교한다.
   - NPC가 러닝트랙/운동기구/책상 위에 떠 있거나 겹치면 실패로 본다.
   - NPC가 관할 공간 안에 들어가야 한다. 공간 밖에 스티커처럼 붙이면 실패로 본다.

3. 각도와 시선은 공간을 따른다.
   - 캐릭터가 바라보는 방향은 홈탭 바닥 원근, 앉은 자리, 사용자가 클릭할 위치, 주변 소품과 맞춘다.
   - 단순히 `scaleX(-1)`로 뒤집기 전에, 뒤집은 결과가 공간 안에서 자연스러운지 미리보기 합성으로 확인한다.

4. 임의 스케일 변경을 금지한다.
   - “대충 작게/크게”가 아니라 기존 NPC/actor와 비교한 기준 px를 기록한다.
   - 홈 스프라이트, 전구, 이름표, 모달 아트는 각각 별도 크기 기준을 갖는다.
   - 화면에서 혼자 튀면 자산 자체를 다시 만들거나 별도 홈용 스프라이트를 만든다.

## 이미지 생성/후처리 원칙

1. imagegen 프롬프트에는 반드시 다음을 포함한다.
   - 홈탭 기존 아이소메트릭 픽셀아트/반픽셀아트 화풍
   - 투명 배경 또는 chroma-key 배경
   - 텍스트/로고/말풍선/이상한 도형 금지
   - 홈 배치용이면 작은 스프라이트 구도, 모달용이면 모달 전용 자세
   - 캐릭터가 서거나 앉을 바닥 방향과 시선

2. 실존 인물/저작물 모티브는 직접 복제가 아니라 분위기 기준으로 쓴다.
   - 예: “검은 선글라스와 나이 든 패션 편집장 분위기의 원본 NPC”
   - 특정 배우/캐릭터의 정확한 얼굴 복제나 로고/상표 재현은 피한다.

3. 도형처럼 보이는 결과는 폐기한다.
   - 캐릭터가 공간 안 물체가 아니라 카드, 스티커, 추상 도형, UI blob처럼 보이면 재생성한다.
   - 생성 결과에 글자가 뭉개져 있으면 이미지 안 텍스트를 쓰지 않고 DOM 텍스트로 처리한다.

4. PNG 후처리 기준
   - 최종 자산은 RGBA 투명 PNG여야 한다.
   - 알파 가장자리, 잔여 chroma-key, 깨진 배경, 잘린 팔다리를 확인한다.
   - PNG header 크기와 `colorType: 6`을 테스트에 넣는다.

## UI/이벤트 원칙

1. NPC 이름표는 DOM 텍스트로 렌더한다.
   - 홈탭 NPC 이름은 트레이너처럼 `lz-nameplate--npc` 계열로 표기한다.
   - 이미지에 이름을 굽지 않는다.

2. 전구 상호작용은 공통 구조를 따른다.
   - 전구는 기존 `npc-quest-bubble.png`를 우선 재사용한다.
   - 클릭 가능한 요소는 `button`이어야 하고, `aria-label`, `title`, `data-lz-action`을 명확히 둔다.
   - 이벤트는 `life-zone:npc-quest`를 유지하고 `detail: { npc: '<id>' }`로 분기한다.

3. 모달 내부 버튼은 직접 바인딩한다.
   - 모달 sheet가 `event.stopPropagation()`을 쓰면 delegated handler에 기대지 않는다.
   - inline `onclick`을 새로 만들지 않는다.

## 구현 전 체크리스트

NPC 요청이 들어오면 앱 코드 수정 전에 다음을 답해야 한다.

1. 이 NPC의 홈 위치는 기존 방의 어느 좌표/공간인가?
2. 같은 공간에 이미 있는 가구/운동기구/캐릭터와 겹치는가?
3. 홈 배치용 스프라이트와 모달용 아트에셋을 각각 만들었는가?
4. NPC 전용 공간/소품 overlay가 필요한가?
5. 캐릭터 시선이 홈탭 원근과 맞는가?
6. 스프라이트 크기가 기존 NPC/actor와 비교해 자연스러운가?
7. 이름표와 전구는 이미지가 아니라 DOM 구조로 들어가는가?
8. 새 PNG/JS/CSS가 `sw.js` `STATIC_ASSETS`에 포함되는가?
9. `CACHE_VERSION` bump가 필요한 파일을 건드렸는가?
10. 배포 후 인증 세션 없이는 홈 UI flow가 막히는지, 막히면 어떤 검증까지 가능한지 기록할 것인가?
