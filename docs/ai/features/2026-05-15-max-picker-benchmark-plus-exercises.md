# Max 테스트모드 종목 추가 피커: 벤치마크 + 같은 부위 종목

## 문제

- 테스트모드 운동 추가 피커가 벤치마크 resolver 결과가 있으면 그 목록만 반환한다.
- 그래서 오늘 벤치마크와 같은 운동만 사실상 강제되고, 같은 부위에 등록된 랫풀다운/로우/컬 같은 추가 종목을 바로 넣을 수 없다.

## 목표

- 벤치마크 종목은 계속 최상단 우선 후보로 유지한다.
- 선택한 테스트모드 부위와 같은 등록 운동도 피커에 함께 노출한다.
- 현재 헬스장 또는 공통 종목만 추가 후보로 넣어, 다른 헬스장 전용 종목이 섞이지 않게 한다.

## 그릴 결과

- 핵심 질문: 테스트모드에서 벤치마크 계획값을 잃지 않으면서 다른 종목도 넣어야 하는가?
- 결정: 벤치마크 항목은 기존 계획 세트 생성 로직을 그대로 쓰고, 추가 종목은 일반 수동 추가 항목으로 만든다.
- 남은 가정: 다른 헬스장 전용 종목은 실제 수행 가능성이 낮으므로 기본 후보에서 제외한다.

## 실행 슬라이스

### Slice 1: 피커 후보 확장

- `resolveMaxBenchmarkPickerItems()`가 벤치마크 항목 뒤에 같은 부위의 등록 운동을 추가한다.
- 추가 후보는 `muscleId`, `muscleIds`, `movementId`를 기준으로 대분류 부위를 판정한다.
- `currentGymId`가 있으면 공통 또는 현재 헬스장 종목만 포함한다.

### Slice 2: UI 문구 및 회귀 테스트

- 테스트모드 피커 안내 문구를 “벤치마크 + 같은 부위 추가 종목”으로 바꾼다.
- 랫풀다운 같은 같은 부위 종목이 벤치마크가 아니어도 표시되는 회귀 테스트를 추가한다.

### Slice 3: 버전/검증/리뷰

- 변경 정적 자산 쿼리와 service worker cache version을 갱신한다.
- `node --check`, `node --test tests/calc.max.test.js`, `git diff --check`를 통과시킨다.

## 비범위

- 계획 조정 모달의 벤치마크 선택 목록 SSOT는 이전 슬라이스 범위로 유지한다.
- W1 시작일 이전 기록을 그래프에 별도 표시하는 작업은 이번 슬라이스에 포함하지 않는다.
