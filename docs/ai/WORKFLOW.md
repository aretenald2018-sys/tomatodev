> 세션 분리와 "세션당 슬라이스 1개" 강제를 모든 변경에 요구했다. 실제 커밋 이력을 보면

> 이 프로젝트는 크고 연관된 변경을 실제로 한 번에 묶어서 처리해 왔고, `AGENTS.md`(Codex 규칙)도
> 처음부터 "기본적으로 바로 구현, 계획은 필요할 때만"이라는 반대 철학을 명시하고 있었다.
> 그 불일치를 정리하고 실제 practice에 맞춰 이 문서를 가볍게 다시 썼다.

## 기본 원칙

- 기본적으로 바로 구현한다. 계획 문서(`docs/ai/features/*.md`)는 다음 중 하나에 해당할 때만 먼저 쓴다:
  - 요청 범위가 실질적으로 모호해서 구현 방향을 먼저 정해야 할 때
  - 변경이 독립적인 여러 모듈/파일에 걸쳐 있어 순서와 경계를 미리 정리하는 게 나을 때
  - 사용자가 명시적으로 계획을 요청했을 때
- 코드에서 답을 알 수 있는 질문은 사용자에게 묻지 말고 먼저 탐색한다.
- 계획 문서를 쓰기로 했어도 이후 실행은 하나의 이어지는 작업으로 본다. "planning session /
  execution session / review session"처럼 경직된 세션 종류를 강제하지 않고, "세션당 정확히
  슬라이스 1개"로 인위적으로 쪼개지 않는다. 승인된 범위 안에서 안전하게 리뷰 가능한 만큼은
  한 번에 구현한다.

## 문서 언어 규칙

- AI가 생성하는 계획, 리뷰, ADR, 로드맵, 핸드오프 문서는 기본적으로 한국어로 작성한다.
- 코드 식별자, 파일 경로, 명령어, API 이름, 라이브러리 이름, 환경 변수 이름, 인용 원문은 원래 언어를 유지한다.
- 사용자가 특정 산출물에 대해 다른 언어를 명시적으로 요청한 경우에만 그 산출물에 한해 따른다.

## 버그 진단이 유용한 경우

버그가 재현이 안 되거나 원인이 불확실할 때는 추측으로 바로 고치기보다 짧은 진단을 먼저 한다.
모든 버그에 의무는 아니고, 막혔을 때 쓰는 도구다.

- 재현/피드백 루프부터 만든다 — 테스트, CLI 스크립트, fixture, HTTP 확인, 브라우저 확인 중 가장 빠르고 믿을 수 있는 것.
- 사용자가 말한 증상과 같은 실패인지 먼저 확인한다.
- 반증 가능한 원인 가설을 몇 개 세우고 우선순위를 매긴다.
- 계측은 가설 검증에 필요한 최소 범위만 넣고, 임시 로그는 고유 태그를 붙여 마지막에 제거한다.
- 수정 후 원래 재현 루프와 회귀 검증을 다시 실행한다.
- 진단이 길고 복잡했다면 `docs/ai/diagnoses/YYYY-MM-DD-short-slug.md`에 한국어로 남겨도 좋다 (의무는 아님).

## 모호한 설계 질문에 유용한 경우

기능/디자인/UX처럼 답이 여러 갈래로 갈리는 요청은, 구현 전에 중요한 질문 한두 개만 물어서
방향을 정하는 게 나을 수 있다. 이때도:

- 한 번에 하나의 중요한 질문만 한다. 질문마다 추천 답변을 함께 제시한다.
- 코드베이스를 보면 답할 수 있는 질문은 묻지 않는다.
- 이건 "모든 기능/디자인 요청에 자동 적용해야 하는 게이트"가 아니라, 판단이 필요할 때 쓰는 방법이다.

## NEXT_ACTION.md — 선택적 인계 메모

`docs/ai/NEXT_ACTION.md`는 여러 세션에 걸치는 크고 복잡한 작업에서만 선택적으로 쓰는
핸드오프 메모다. 매 요청, 매 커밋마다 의무적으로 갱신하지 않는다.

- 다음 세션에서 이어받아야 할 맥락이 있는 큰 작업이면, 상태/계획 경로/막힌 지점을 간단히 남긴다.
- 작업이 그 세션 안에서 끝나면 굳이 남기지 않아도 된다.

## Context Hygiene

- 채팅 기억보다 durable한 문서가 유용할 때가 많다 — 다만 이것도 의무가 아니라, 다음에 이어받을
  사람(자신 포함)이 맥락 없이 이해해야 하는 크고 복잡한 작업에서 판단해서 남긴다.
- `docs/ai/CONTEXT.md`는 프로젝트 용어/도메인 가정 참고용으로 계속 쓴다.
- `docs/adr/`는 되돌리기 비싼 결정을 기록하는 용도로 계속 쓴다.
