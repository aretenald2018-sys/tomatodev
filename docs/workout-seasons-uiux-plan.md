# 운동·식단 시즌제 및 운동 탭 UI/UX 통합 계획

## 1. 목표

사용자가 잘못 입력했거나 빠뜨린 과거 기록 때문에 현재 목표, 추천, 직전 기록, 성장 추이, 식단 성공률이 계속 왜곡되지 않도록 `시즌`을 계산 경계로 도입한다.

- 이전 운동·식단 기록은 삭제하거나 덮어쓰지 않는다.
- 새 시즌의 목표와 계획은 새 시즌 데이터만 사용한다.
- 지난 시즌은 달력에서 계속 열람하고 수정할 수 있다.
- 지난 시즌 수정은 그 시즌의 통계만 바꾸며 현재 시즌의 추천·목표에는 영향을 주지 않는다.
- 운동 탭의 달력, 날짜 상세, 운동 카드, 종목 선택, 설정 모달은 하나의 시각·상호작용 규칙을 사용한다.

## 2. 현재 체크아웃에서 확인한 상태

현재 코드에는 이미 시즌 기능의 1차 기반이 들어가 있다.

- `data/season-model.js`: 시즌 날짜 범위와 레지스트리 모델
- `data/data-api.js`: 활성 시즌, 시즌별 식단 플랜·성장 보드·MAX 사이클, 시즌별 목표
- `data/data-load.js`: 기존 계정을 최초 시즌으로 옮기는 로드 시 마이그레이션
- `render-calendar.js`: 날짜별 시즌 색, 시즌 시작 경계, 시즌 칩
- `modals/settings-modal.js`: 새 시즌 시작 UI
- `tests/season-model.test.js`, `tests/season-integration.test.js`: 기본 보존·경계 계약

따라서 본 계획은 이 기반을 버리지 않고, 데이터 누수 지점을 막고 전환 UX와 운동 탭 디자인을 완성하는 방향으로 진행한다.

### 현재 기반에서 보완해야 할 핵심

1. `workout/expert.js`, `workout/expert/max.js` 등 일부 의사결정 경로가 아직 전체 `getCache()`를 직접 읽는다. 과거 시즌 기록이 현재 처방에 섞일 수 있다.
2. 통계의 기본 기간은 `이번 시즌`이지만 일부 하위 계산과 내보내기는 전체 캐시를 사용한다. `현재 시즌 통계`와 `전체 기록 보기`를 명시적으로 분리해야 한다.
3. 시즌 전환이 `writeBatch` 기반이라 여러 기기에서 동시에 시작할 때 마지막 쓰기가 앞선 전환을 덮을 수 있다. 트랜잭션과 중복 요청 방지가 필요하다.
4. 달력의 월 요약은 한 달에 두 시즌이 섞여도 월 전체를 합산한다. 선택한 날짜의 시즌 요약과 월 전체 요약이 구분되어야 한다.
5. 시즌 관리는 일반 설정 모달 안에 섞여 있다. 운동·식단의 핵심 맥락이므로 운동/식단 헤더의 시즌 칩에서 전용 시트를 여는 편이 자연스럽다.
6. 실제 화면에서 달력은 회색 배경·8px 카드, 날짜 상세은 각진 대형 시트, 종목 선택은 순백색 풀스크린·빨간 탭을 사용한다. 같은 작업 흐름인데 표면색, 모서리, 강조색, 헤더 높이, 버튼 문법이 연속적으로 바뀐다.

## 3. 제품 규칙

### 3.1 데이터 보존과 계산 경계를 분리한다

- 원본 기록: 날짜 문서가 단일 진실이다. 삭제하거나 새 컬렉션으로 복사하지 않는다.
- 계산 범위: 시즌의 `startDate`와 `endDate`로 읽을 날짜만 제한한다.
- 목표/계획: 시즌에 직접 귀속되는 설정은 `seasonId`를 가진다.
- 전역 자산: 운동 종목 목록, 헬스장, 기구, 음식 DB는 시즌이 바뀌어도 유지한다.

### 3.2 지난 시즌은 보관함이 아니라 계속 편집 가능한 기록이다

- 달력에서 지난 시즌 날짜를 누르면 당시 목표·식단 플랜·운동 프로그램 기준으로 상세를 보여준다.
- 지난 기록을 수정해도 현재 시즌의 직전 기록, PR, 추천, 스트릭, 목표 진척도는 다시 계산하지 않는다.
- 전체 통계와 전체 내보내기에서만 여러 시즌을 함께 볼 수 있다.

### 3.3 시즌 경계는 색만으로 전달하지 않는다

- 날짜 배경에 아주 약한 시즌 톤을 적용한다.
- 시즌 시작일에는 세로 경계선과 `새 시즌` 라벨을 함께 표시한다.
- 헤더와 날짜 상세에 시즌 이름을 텍스트로 표시한다.
- 스크린리더용 날짜 라벨에도 시즌 이름과 시작 여부를 포함한다.

## 4. 권장 데이터 모델

기존 Firestore 경로를 유지한다.

```text
settings/season_registry
  value:
    schemaVersion: 1
    activeSeasonId: season_20260713_xxx
    seasons:
      - id
      - name
      - startDate
      - endDate
      - status
      - tone
      - createdAt
      - closedAt

settings/season_<seasonId>_diet_plan
settings/season_<seasonId>_unit_goal_start
settings/season_<seasonId>_max_cycle
settings/season_<seasonId>_max_cycle_history
settings/season_<seasonId>_test_board_v2

goals/{goalId}
  seasonId: season_20260713_xxx

workouts/{YYYY-MM-DD}
  기존 운동·식단 필드 유지
```

일별 문서에 `seasonId`를 강제로 저장하지 않는다. 시즌 시작일을 과거 날짜로 지정했을 때 문서를 일괄 수정하지 않아도 되고, 날짜 범위라는 한 가지 기준으로 일관되게 분류할 수 있기 때문이다. 시즌이 생성된 뒤에는 시작일과 과거 시즌 경계를 수정하지 않는 것을 원칙으로 한다.

### 데이터 범위 표

| 데이터 | 시즌 전환 시 | 기본 조회 범위 |
|---|---|---|
| 운동·러닝·식단 일별 기록 | 보존 | 선택한 날짜가 속한 시즌 |
| 목표 | 이전 시즌에서 종료 | 활성 시즌 |
| 식단 목표/허용 범위 | 새로 설정 | 선택한 날짜가 속한 시즌 |
| MAX 사이클·성장 보드 | 새로 시작 | 활성 시즌 |
| 운동 종목·헬스장·기구 | 유지 | 전역 |
| 체중·체지방 원본 | 유지 | 전역 보관, 차트 기본은 활성 시즌 |
| 퀘스트 정의 | 유지 | 진행도 계산은 활성 시즌 시작일 이후 |
| 스트릭·추천·PR·직전 기록 | 재시작 | 활성 시즌 |
| 전체 통계·CSV | 보존 | 사용자가 `전체`를 명시한 경우만 전 시즌 |

## 5. 시즌 전환 처리

### 5.1 시작 전 미리보기

사용자가 이름과 시작일을 정하면 저장 전에 다음을 보여준다.

- 종료되는 현재 시즌과 종료일
- 새 시즌으로 분류될 기존 기록 일수(과거 시작일을 선택한 경우)
- 종료되는 목표 수
- 초기화되는 항목: 목표, 식단 플랜, 성장 보드, MAX 사이클, 스트릭
- 유지되는 항목: 원본 기록, 종목, 헬스장, 기구, 체중 기록, 사진

### 5.2 원자적 전환

`data.js` 경로 안에서 Firestore 트랜잭션으로 처리한다.

1. 최신 `season_registry`를 다시 읽는다.
2. 요청한 시작일이 최신 활성 시즌 시작일보다 뒤이고 오늘보다 늦지 않은지 검증한다.
3. 현재 시즌의 `endDate`를 새 시작일 전날로 닫는다.
4. 새 시즌과 초기 설정 문서를 한 트랜잭션에서 만든다.
5. 성공 후에만 메모리 상태를 교체하고 `season:changed`를 발행한다.
6. 클라이언트 요청 ID를 두어 이중 탭·재시도로 같은 시즌이 두 번 생기지 않게 한다.

원본 운동·식단 문서는 이 과정에서 읽기만 하며 삭제·수정하지 않는다.

### 5.3 마이그레이션

- 시즌 레지스트리가 없는 계정은 가장 오래된 날짜 기록을 시작일로 하는 `기존 시즌`을 만든다.
- 시즌 ID가 없는 기존 목표는 최초 시즌에만 귀속한다.
- 마이그레이션 완료 마커를 별도로 저장해 매 로드마다 쓰기를 반복하지 않는다.
- 일부 설정 쓰기가 실패해도 다시 실행 가능한 멱등 방식으로 만든다.

## 6. 읽기 경로 정리

`getCache()`는 원본 조회와 전체 내보내기에서만 사용한다. 목표·추천·처방에 쓰는 코드는 다음 중 하나를 명시해야 한다.

```js
getSeasonScopedCache()                // 활성 시즌
getSeasonScopedCache(dateKey)         // 해당 날짜가 속한 시즌
getSeasonScopedCache(seasonId)         // 지정 시즌
getCache()                             // 전체 기록, 명시적 전체 보기만
```

### 반드시 감사할 경로

- 직전 세트와 추천 시작 무게
- 운동 PR, 볼륨/강도 스파크라인
- 전문가 모드와 MAX 처방
- 성장 보드 후보와 최근 기록
- 운동·식단 AI 추천
- 스트릭과 퀘스트 자동 진행도
- 통계의 기간별 비교와 칼로리/체중 차트
- 홈의 목표 진척도와 단위 목표

각 경로는 `decisionScope: season | all-time` 계약을 테스트로 고정한다.

## 7. 시즌 UX

### 진입점

- 1차: 운동 탭과 식단 탭 헤더의 시즌 칩
- 2차: 더보기의 `시즌 관리`
- 일반 설정 모달에서는 시즌 관리 링크만 제공한다.

### 전용 시즌 시트

1. 현재 시즌 요약
2. 이전 시즌 타임라인
3. `새 시즌 시작` 버튼
4. 이름·시작일 입력
5. 변경 영향 미리보기
6. 마지막 확인

파괴적 삭제가 아니므로 빨간 경고 화면은 쓰지 않는다. 대신 `원본 기록은 그대로 유지됩니다`를 가장 먼저 보여주고, 초기화되는 계산 항목을 명확히 적는다.

### 달력과 목표

- 헤더 칩: 선택한 날짜가 속한 시즌 이름
- 달력 셀: 시즌별 4~6% 정도의 미묘한 배경 톤
- 경계일: 2px 세로선 + `새 시즌`
- 월 요약: 기본은 선택한 날짜의 시즌과 겹치는 날짜만 합산하고 `여름 리셋 · 7월`처럼 범위를 표시
- 전체 월 합계가 필요하면 사용자가 `7월 전체`를 명시적으로 선택
- 지난 시즌 날짜 상세: `지난 시즌` 배지와 당시 목표·플랜 표시
- 홈 목표: 활성 시즌 목표만 표시하고 시즌 이름을 섹션 헤더 보조 정보로 제공

## 8. 운동 탭 통합 디자인 시스템

문서상의 기존 디자인 시스템보다 실제 운동 흐름의 일관성을 우선한다.

### 시각 토큰

```text
배경       #F2F2F7
주 표면    #FFFFFF
보조 표면  #F7F7FA
주 텍스트  #1C1C1E
보조 텍스트 #6E6E73
구분선     rgba(60, 60, 67, 0.16)
주 액션    #0A84FF
성공       #34C759
주의       #FF9F0A
삭제       #FF453A

카드 반경       16px
입력/작은 버튼  12px
대형 시트 상단  26px
터치 영역       최소 44px
간격 척도       4 / 8 / 12 / 16 / 24 / 32
```

### 공통 컴포넌트

- `AppHeader`: 큰 제목, 보조 문구, 시즌 칩
- `MonthNavigator`: 이전/다음, 월, 오늘
- `MetricStrip`: 3개의 핵심 지표만 같은 형식으로 표시
- `SeasonCell`: 배경 톤, 경계, 상태 점
- `Sheet`: handle + 52px 헤더 + 스크롤 본문 + 안전영역 액션
- `SegmentedControl`: 회차, 러닝, 필터에 같은 형태 사용
- `WorkoutCard`: 제목, 컨텍스트, 핵심 목표, 세트 행, 완료 액션
- `SearchField`: 종목 선택과 기타 선택 모달 공통
- `ListRow`: 아이콘/텍스트/보조 정보/chevron의 iOS형 행
- `PrimaryButton`, `SecondaryButton`, `DestructiveAction`: 색과 위치를 고정

### 상호작용 규칙

- 날짜 상세은 `collapsed → medium → large` detent를 사용한다.
- 종목 선택과 시즌 관리는 같은 `Sheet`의 large 변형을 사용한다.
- 중첩 화면은 기존 시트를 없애지 않고 위에 한 장만 쌓고, 뒤로가기로 원래 스크롤 위치에 복귀한다.
- 내부 컨테이너가 `stopPropagation()`을 쓰더라도 모든 버튼은 시트 내부에 직접 바인딩한다.
- 화면 상시 노출 버튼은 lazy module의 전역 함수에 의존하지 않는다.
- 완료 후에는 시트 전체를 닫지 않고 해당 카드 상태만 갱신해 맥락을 보존한다.

## 9. 구현 순서

### 1단계 — 범위 계약과 데이터 안전성

- 모든 `getCache()` 호출을 `전체 기록`과 `시즌 의사결정`으로 분류
- 전문가 모드, MAX, 통계 하위 계산의 시즌 누수 수정
- 시즌 트랜잭션과 멱등 마이그레이션 추가
- 과거 시작일 선택 시 영향 미리보기 API 추가

### 2단계 — 시즌 UX 완성

- 전용 시즌 관리 시트
- 운동/식단 헤더 시즌 칩
- 지난 시즌 열람 상태와 현재 시즌 복귀 동선
- 월 요약의 시즌 범위 선택
- 목표 모달에 현재 시즌 안내

### 3단계 — 운동 탭 표면 통합

- 토큰과 공통 Sheet/Header/SegmentedControl 적용
- 달력 헤더·요약·셀 재배치
- 날짜 상세과 운동 카드 통합
- 종목 선택, 종목 편집, 세트 편집, 러닝 진입, 목표 입력 모달 통합

### 4단계 — 검증과 배포

- 시즌 단위/통합 테스트
- 기존 기록 보존 전후 스냅샷 비교
- 모바일 운영 화면에서 운동 → 날짜 → 회차 → 운동 추가 → 종목 선택 → 복귀 흐름 실행
- 시즌 관리 → 영향 미리보기 → 시작 → 목표/식단/추천 초기화 확인
- 서비스 워커 정적 자산 변경 시 `CACHE_VERSION` 갱신
- `origin/main` 배포 후 GitHub Pages 커밋 검증

## 10. 완료 조건

1. 시즌 전환 전후 일별 문서 수와 원본 사진 필드가 동일하다.
2. 이전 시즌 기록이 현재 시즌의 직전 기록, 추천, PR, 스트릭, 목표 진척도에 영향을 주지 않는다.
3. 지난 시즌 날짜를 열면 당시 플랜으로 성공 여부와 상세를 재현한다.
4. 한 달에 여러 시즌이 있어도 월 요약의 계산 범위가 화면에 명시된다.
5. 이전 시즌 목표는 보존되지만 홈에는 활성 시즌 목표만 나타난다.
6. 달력, 날짜 상세, 종목 선택, 종목 편집, 시즌 관리가 같은 색·반경·헤더·버튼 규칙을 사용한다.
7. 모든 새 버튼과 탭을 실제 브라우저에서 클릭해 동작과 뒤로가기를 확인한다.

## 11. 목업 범위

`docs/workout-seasons-uiux-mockup.html`은 다음 상태를 한 화면에서 전환해 볼 수 있는 독립 목업이다.

- 시즌 경계가 있는 운동 월간 홈
- 날짜 상세 시트
- 통합된 종목 선택 시트
- 시즌 관리 시트

실제 데이터 저장은 하지 않으며, 최종 구현 전에 정보 밀도·색 대비·시트 계층을 합의하기 위한 시각 기준이다.
