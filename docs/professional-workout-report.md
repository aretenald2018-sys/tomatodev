# 프로페셔널 운동 리포트 기획안

## 1. 목표

관리자가 원하는 기간을 지정하면 운동탭의 일반, 프로, 테스트모드 사용자 모두에게 같은 품질의 프로페셔널 운동 리포트를 발행한다. 현재 통계 탭의 `심층통계`가 가진 유효세트, 체감강도, 최근 2주 변화 계산을 확장하되, 테스트모드의 6주 메조사이클 UX를 일반/프로 사용자에게 그대로 노출하지 않는다.

핵심 결과물은 “관리자 지정 기간에 대한 전문가 리포트”다. 이용자는 내가 어느 부위를 충분히 훈련했고, 어느 부위는 보완해야 하며, 다음 1-2주에 무엇을 조절해야 하는지 바로 이해해야 한다.

## 2. 모드별 리포트 원칙

| 사용자 모드 | 리포트 표현 | 주의점 |
| --- | --- | --- |
| 일반모드 | 관리자 지정 기간 기준 요약 | 고급 용어를 줄이고 행동 제안을 앞에 둔다. |
| 프로모드 | 관리자 지정 기간 + 운동 품질/부하 분석 | RPE, 유효세트, 진행률, ROM 신뢰도까지 보여준다. |
| 테스트모드 | 관리자 지정 기간 + 테스트 사이클 맥락 | 6주 루틴 정보가 있을 때만 `n/6주차`를 보조 정보로 노출한다. |

일반/프로 사용자에게는 `3/6주차` 같은 고정 사이클 문구를 쓰지 않는다. 대신 `이번 4주 요약`, `2026.04.08-2026.05.05`, `관리자 발행 리포트`처럼 선택 기간이 먼저 보이게 한다.

## 3. 관리자 발행 플로우

1. 관리자가 리포트 기간을 선택한다.
2. 대상 사용자를 선택한다. 전체, 그룹, 태그, 개별 사용자 단위가 가능해야 한다.
3. 비교 기준을 선택한다. 기본은 선택 기간을 절반으로 나눈 `최근 절반 vs 이전 절반` 비교다.
4. 시스템이 사용자별 리포트를 생성하고, 관리자에게 미리보기를 제공한다.
5. 관리자가 문구를 검토한 뒤 발행한다.
6. 이용자는 운동탭 상단 또는 통계탭에서 최신 리포트를 확인한다.

추천 기본값:

- 기간: 최근 4주
- 비교: 최근 2주 vs 이전 2주
- 최소 데이터 기준: 운동일 3일 이상 또는 유효세트 20세트 이상
- 데이터 부족 시: 성과 판정 대신 “기록 안정화 리포트”로 발행

## 4. 리포트 카드 구성

### 4.1 히어로 요약 카드

현재 테스트모드의 히어로카드 형태를 유지하되, 문구를 기간 기반으로 바꾼다.

예시:

```text
이번 4주 요약
쌓는 구간
체감강도 평균 8.6 · 주당 유효세트 103 · 최근 2주 -23세트 / +2일
```

상태 라벨은 고정 주차가 아니라 데이터로 추론한다.

| 라벨 | 조건 예시 | 의미 |
| --- | --- | --- |
| 쌓는 구간 | 유효세트가 유지 또는 증가, RPE가 통제됨 | 훈련 자극을 누적해도 되는 상태 |
| 회복 필요 | 고RPE 비율 증가, 세트 수 과다, ROM 저하 | 다음 주 볼륨이나 강도 조정 필요 |
| 재정렬 구간 | 운동일은 늘었지만 핵심 부위 자극이 낮음 | 루틴 구성 수정 필요 |
| 기록 안정화 | 데이터가 부족하거나 누락이 많음 | 판단보다 기록 습관 형성 우선 |

### 4.2 신체 부위 활성 지도

두 번째 카드는 사람 신체 모델을 중심으로 둔다. 수행한 운동의 부위별 유효세트, RPE, ROM, 진행률을 합쳐 부위 상태를 색으로 표시한다.

렌더링은 운동기구를 들고 있는 포즈가 아니라 빈손의 해부학 라인아트 인체를 기본으로 한다. 근섬유 결을 얇은 선으로 표현하고, 그 위에 데이터 기반 활성 색상을 얹는다. 이렇게 해야 운동 종류와 무관하게 같은 모델을 재사용할 수 있고, 사용자별/기간별 리포트에서 색상만 바꿔도 일관된 신체 지도가 된다.

색상 의미:

- 파랑: 미활성 또는 최소 유효 볼륨 미달
- 회색: 기록 부족 또는 판정 보류
- 옅은 토마토: 적정 자극 범위
- 진한 토마토: 높은 자극이지만 관리 가능한 범위
- 빨강: 과활성, 피로 누적, 또는 회복 리스크

이 카드는 단순한 “잘함/못함” 판정이 아니라 넛징이어야 한다.

예시 문구:

- 가슴/삼두는 충분히 잘 쌓이고 있어요. 다음 1주 동안은 세트 수를 더 늘리기보다 마지막 2세트의 ROM을 유지해보세요.
- 햄스트링과 후면 어깨는 상대적으로 비어 있습니다. 다음 하체일에 힙힌지 3-4세트, 등 운동일에 리어델트 2-3세트를 추가하면 균형이 좋아집니다.

### 4.3 부위별 볼륨 랜드마크

부위별 주당 유효세트를 최소 유효 볼륨, 적정 볼륨, 과부하 리스크 구간과 비교한다.

초기 기준값은 현재 심층통계에서 쓰는 랜드마크를 확장한다.

| 부위 | 낮음 | 적정 | 높음 |
| --- | ---: | ---: | ---: |
| 가슴 | 8 | 14 | 22 |
| 등 | 10 | 16 | 25 |
| 하체 | 8 | 14 | 20 |
| 어깨 | 6 | 14 | 22 |
| 이두 | 6 | 12 | 20 |
| 삼두 | 6 | 14 | 18 |
| 복근 | 0 | 12 | 25 |

운동 카탈로그가 세분화되면 하체를 둔근, 대퇴사두, 햄스트링, 종아리로 나누고 어깨를 전면, 측면, 후면으로 나눈다. 세분화 매핑이 없으면 기존 major muscle 기준으로 먼저 계산한다.

### 4.4 진행률과 정체 신호

주요 운동별 예상 1RM, 동일 중량 반복수, RPE 대비 수행량 변화를 본다.

분석 예시:

- 벤치프레스: 예상 1RM +2.4kg, RPE는 동일. 좋은 진행.
- 랫풀다운: 중량 변화는 없고 RPE만 +1.1 상승. 피로 또는 기술 효율 저하 가능.
- 스쿼트: 최근 3회 노출에서 총 반복수 정체. 다음 2주간 세트 수보다 반복 품질 우선.

정체 판정은 최소 3회 이상 노출된 운동에만 적용한다. 1-2회 기록으로는 “추세 관찰”만 표시한다.

### 4.5 피로와 회복 카드

운동생리학 관점에서 단기 성과보다 회복 가능성을 같이 본다.

주요 지표:

- 평균 RPE
- RPE 9 이상 세트 비율
- 최근 절반의 유효세트 변화
- 운동일 변화
- 같은 운동에서 ROM 저하와 RPE 상승이 동시에 발생했는지
- 고중량 복합운동이 연속으로 몰렸는지

제안 예시:

- 고RPE 세트 비율이 28%입니다. 다음 1주일은 실패지점 근처 세트를 줄이고 RPE 7-8 세트를 늘리는 편이 회복 대비 효율이 좋습니다.
- 운동일은 2일 늘었지만 유효세트는 23세트 줄었습니다. 빈도는 좋아졌으니 세션당 핵심 운동 1개만 안정적으로 고정하세요.

### 4.6 ROM과 수행 품질 카드

테스트모드에서 세트별 가동범위 퍼센트를 저장할 수 있게 되면, 이 리포트는 ROM 신뢰도를 강하게 활용한다.

분석 원칙:

- ROM이 줄었는데 중량만 오른 PR은 보수적으로 해석한다.
- ROM이 안정된 상태에서 반복수나 중량이 오른 경우를 “신뢰도 높은 진행”으로 표시한다.
- 동일 운동의 ROM 변동폭이 크면 기술 안정화 제안을 먼저 낸다.

예시:

- 인클라인 프레스의 최근 평균 ROM이 82%입니다. 중량 증가는 있었지만 가슴 자극 신뢰도는 중간입니다.
- 루마니안 데드리프트는 ROM 95% 이상을 유지하면서 반복수가 증가했습니다. 햄스트링 진행 신뢰도가 높습니다.

### 4.7 균형과 상쇄 카드

부위별 총량만 보지 않고 길항 관계와 패턴 균형을 본다.

기본 비교:

- 밀기 vs 당기기
- 수직 당기기 vs 수평 당기기
- 대퇴사두 중심 vs 힙힌지/햄스트링 중심
- 전면 어깨 vs 후면 어깨
- 이두 vs 삼두
- 좌우 편측 운동의 볼륨 차이

예시:

- 밀기 세트가 당기기보다 1.7배 많습니다. 어깨 전면 피로가 쌓이기 쉬우니 수평 당기기와 후면 어깨를 보강하세요.
- 하체 볼륨은 충분하지만 햄스트링 직접 자극이 낮습니다. 레그컬 또는 RDL을 하체일 초반에 배치하세요.

### 4.8 영양과 체성분 컨텍스트

운동 리포트는 식단 데이터를 필수로 요구하지 않지만, 존재하면 해석에 반영한다.

사용 가능한 데이터:

- 체중 추세
- 섭취 칼로리 평균
- 단백질 g/kg
- 운동일과 비운동일 섭취 차이

예시:

- 체중은 유지인데 주당 유효세트가 높습니다. 근비대 목적이라면 단백질과 총칼로리의 하한선을 먼저 안정화하세요.
- 체중이 빠지는 중에도 RPE가 상승하고 있습니다. 감량 폭보다 훈련 품질 유지가 우선입니다.

### 4.9 다음 2주 처방 카드

리포트의 마지막 핵심은 “그래서 무엇을 할지”다.

출력 형식:

| 영역 | 처방 | 이유 |
| --- | --- | --- |
| 볼륨 | 가슴 -4세트, 등 +3세트 | 밀기 과다와 당기기 부족 보정 |
| 강도 | 복합운동 마지막 세트 RPE 8 제한 | 고RPE 누적 완화 |
| 운동 선택 | 리어델트 2세트, 레그컬 3세트 추가 | 미활성 부위 보완 |
| 기술 | 벤치 ROM 90% 이상 유지 | 진행 신뢰도 확보 |

일반모드 사용자는 이 표를 더 쉽게 번역한 “다음 운동에서 할 일” 카드로 볼 수 있다.

## 5. 계산 설계

### 5.1 기간 정규화

관리자가 지정한 시작일과 종료일을 기준으로 분석한다.

```text
days = toDate - fromDate + 1
periodWeeks = days / 7
weeklyHardSets = hardSets / periodWeeks
displayWeeks = round(periodWeeks)
```

표시 문구는 `이번 {displayWeeks}주 요약`을 기본으로 한다. 기간이 10일처럼 애매하면 `최근 10일 요약`으로 표시한다.

### 5.2 유효세트

현재 심층통계의 hard set 정의를 유지한다.

- 워밍업 세트 제외
- 완료되지 않은 세트 제외
- 중량과 반복수가 모두 있는 세트
- RPE가 있으면 RPE 7 이상
- RPE가 없으면 반복수 5회 이상을 임시 기준으로 사용

추후에는 운동별 특성에 따라 보정한다. 예를 들어 측면 어깨 레이즈는 낮은 중량 고반복도 유효세트로 봐야 한다.

### 5.3 비교 기준

기본 비교는 선택 기간을 절반으로 나누는 방식이다.

```text
priorWindow = 선택 기간의 앞 절반
recentWindow = 선택 기간의 뒤 절반
deltaHardSets = recentWindow.hardSets - priorWindow.hardSets
deltaTrainingDays = recentWindow.trainingDays - priorWindow.trainingDays
```

관리자 옵션으로 `직전 동일 기간` 비교도 제공할 수 있다.

### 5.4 신체 지도 점수

부위별 색상은 볼륨과 피로를 함께 본다.

```text
loadRatio = weeklyHardSets / goodLandmark
underFlag = weeklyHardSets < lowLandmark
overFlag = weeklyHardSets > highLandmark
highRpeShare = rpe9PlusSets / hardSets
rpeDrift = recentAvgRpe - priorAvgRpe
romDrop = priorAvgRom - recentAvgRom

fatigueRisk = 0.4 * overFlag
            + 0.25 * highRpeShare
            + 0.2 * positiveRpeDrift
            + 0.15 * positiveRomDrop
```

색상 판정:

| 상태 | 조건 |
| --- | --- |
| 파랑 | `weeklyHardSets < lowLandmark` |
| 회색 | 데이터 부족 |
| 적정 토마토 | `lowLandmark <= weeklyHardSets <= goodLandmark` |
| 진한 토마토 | `goodLandmark < weeklyHardSets <= highLandmark` 이고 피로 리스크 낮음 |
| 빨강 | `weeklyHardSets > highLandmark` 또는 피로 리스크 높음 |

빨강은 “나쁜 부위”가 아니라 “자극과 피로가 높아 관리해야 하는 부위”로 설명한다.

### 5.5 운동별 진행률

근력성 운동은 예상 1RM을 사용한다.

```text
e1RM = weight * (1 + reps / 30)
```

운동별 최근 3회 이상 노출된 기록만 정체/진행 판정에 사용한다. 머신, 케이블, 덤벨, 바벨은 같은 movementId로 묶되, 장비별 중량 호환이 어려운 경우 별도로 표시한다.

### 5.6 데이터 부족 처리

아래 중 하나에 해당하면 과한 판정을 피한다.

- 선택 기간 운동일 3일 미만
- 전체 유효세트 20세트 미만
- 운동 카탈로그의 부위 매핑이 부족한 경우
- RPE와 ROM이 대부분 비어 있는 경우

이때는 “판정”보다 “기록 품질 개선”과 “다음 주 최소 행동”을 제안한다.

## 6. 데이터 구조 초안

Firebase 직접 호출은 금지하고, 구현 시 반드시 `data.js`를 통해 접근한다.

```js
professional_reports/{reportId} = {
  id,
  createdBy,
  audienceType: 'all' | 'group' | 'tag' | 'users',
  targetUserIds: [],
  fromKey: '2026-04-08',
  toKey: '2026-05-05',
  compareMode: 'split-period' | 'previous-equal-period',
  status: 'draft' | 'published' | 'archived',
  generatedAt,
  publishedAt,
  summary: {
    displayTitle: '이번 4주 요약',
    phaseLabel: '쌓는 구간',
    avgRpe: 8.6,
    weeklyHardSets: 103,
    recentDeltaHardSets: -23,
    recentDeltaTrainingDays: 2
  },
  regions: {
    chest: { status: 'high', weeklyHardSets: 24, rpe: 8.9, nudges: [] },
    hamstring: { status: 'low', weeklyHardSets: 3, rpe: 7.4, nudges: [] }
  },
  cards: [],
  adminNote: ''
}
```

사용자별 읽기 최적화가 필요하면 `users/{uid}/professional_reports/{reportId}`에 발행본 스냅샷을 복사한다.

## 7. 관리자 화면 요구사항

관리자 화면에는 다음이 필요하다.

- 기간 선택
- 대상 선택
- 비교 기준 선택
- 리포트 생성 버튼
- 사용자별 데이터 충분/부족 상태
- 발행 전 미리보기
- 관리자 메모
- 발행 상태

발행 전 검토에서 반드시 보여줄 것:

- 데이터 기준을 충족했는지
- 빨강 부위가 과활성인지, 회복 리스크인지
- 파랑 부위가 실제 미훈련인지, 매핑 부족인지
- 이용자에게 노출될 최종 문구

## 8. 이용자 화면 요구사항

운동탭 진입 시 최신 발행 리포트가 있으면 상단에 호출한다.

우선순위:

1. 미확인 최신 리포트
2. 현재 주차/기간과 관련된 리포트
3. 지난 리포트 요약 링크

운동 기록 중에는 리포트가 방해되지 않게 접힌 카드로 시작한다. 다만 “다음 운동에서 할 일”은 운동종목 선택 상단에 짧게 노출할 수 있다.

## 9. 구현 단계 제안

### 1단계: 리포트 생성 엔진

- 현재 `_analyzeTrainerWindow()`를 기간 파라미터 기반으로 분리
- 선택 기간 분석, 최근 절반 비교, 부위별 주당 세트 계산
- 테스트모드 전용 6주 문구 제거

### 2단계: 정적 리포트 UI

- 히어로 요약 카드
- 신체 부위 활성 지도
- 볼륨 랜드마크 표
- 다음 2주 처방 카드

### 3단계: 관리자 발행 UX

- 관리자 기간 선택
- 대상 선택
- 사용자별 미리보기
- 발행 상태 저장

### 4단계: 운동탭 상단 호출

- 최신 발행 리포트 배너
- 운동종목 선택 상단의 “다음 운동 제안”
- 일반/프로/테스트모드별 문구 차등

### 5단계: 고급 분석 확장

- ROM 신뢰도
- 운동별 e1RM 추세
- 길항 균형
- 영양/체성분 컨텍스트

## 10. 수용 기준

- 관리자가 임의 기간을 선택해 리포트를 생성할 수 있다.
- 일반/프로 사용자의 히어로카드에는 `n/6주차` 문구가 나오지 않는다.
- 테스트모드 사용자는 사이클 데이터가 있을 때만 주차 맥락을 보조로 볼 수 있다.
- 신체 모델에서 과활성/관리 필요 부위는 빨강, 미활성/보완 필요 부위는 파랑으로 보인다.
- 리포트는 “분석”에서 끝나지 않고 다음 1-2주 행동 처방을 제공한다.
- 데이터 부족 사용자는 무리한 판정을 받지 않고 기록 안정화 안내를 받는다.
- 구현 시 Firestore 접근은 모두 `data.js`를 통해 이뤄진다.
